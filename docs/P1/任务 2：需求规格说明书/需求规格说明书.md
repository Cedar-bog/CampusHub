## 1 引言

### 1.1 目的

本文档旨在定义“校园互助服务平台”的软件需求规格，为项目的设计、开发、测试和验收提供统一、明确的依据。

### 1.2 范围

本产品是一个面向在校大学生的校园互助服务网站（含移动端浏览器适配访问），旨在整合校内各类碎片化互助需求，提供统一、可信、高效的信息发布与对接渠道。

平台的核心服务范围包括：

- **用户身份认证**：提供基于手机号的快速注册方式，同时将校园统一身份认证作为可选或辅助的实名核验手段，而非强制准入条件。用户在完成实名核验后方可发布需求或接单，以兼顾安全性与用户隐私顾虑。

- **互助业务类型**：涵盖快递代取、跑腿代办、学习辅导与知识分享、二手闲置交易、组队匹配、活动搭伴、失物招领、拼车/拼单等高频校园场景需求。

- **匿名发布支持**：在二手交易、树洞倾诉等特定分类下，允许用户选择匿名方式发布内容，以保护隐私并鼓励自由表达。

- **需求与订单管理**：提供需求的发布、浏览、筛选、接单功能，支持订单状态跟踪及历史记录查询。

- **即时通讯**：供需双方可通过内置聊天工具发送文字、图片消息，满足任务细节沟通需求。

- **信用评价体系**：任务完成后双方可互评打分，系统综合计算形成用户信用参考指标，并予以公开展示。

- **管理员后台**：提供内容审核、用户账号管理、纠纷仲裁与数据统计看板功能。

在**地理范围**方面，平台以**学校**为单位进行实例部署，服务对象限定为本校已注册学生。对于存在多个地理上分离校区的大学，系统支持用户在发布需求时标注具体校区位置，接单者亦可依据校区进行筛选，从而满足实际运作诉求。

第一阶段**明确不包含**的功能包括：在线支付与资金托管、基于实时经纬度的智能距离排序。

### 1.3 定义

本小节对文档中频繁出现的核心术语、缩略语进行统一解释，以避免理解偏差。

| 术语 / 缩略语 | 定义                                                                                                                                  |
| :------------ | :------------------------------------------------------------------------------------------------------------------------------------ |

| **MVP**       | Minimum Viable Product，最小可行产品。指本平台第一阶段的发布版本，仅包含最核心、验证价值最高的功能集合。                              |
| **实名核验**  | 用户通过提交学号/工号并由校园统一身份认证系统验证的过程，用以确认用户的在校学生身份。本平台将实名核验作为可选环节，以平衡安全与隐私。 |
| **匿名发布**  | 用户在特定分类下发布内容时，可选择隐藏自身身份信息，仅显示“匿名用户”标识。                                                            |
| **信用评分**  | 系统基于用户历史任务完成率、双方互评分数等数据综合计算得出的分值，用于直观反映用户的可信程度。                                        |

| **拼车/拼单** | 互助业务类型之一，供用户发起顺风车同行或团购拼单需求，以分摊成本或结伴出行。                                                          |

## 2 总体描述

### 2.1 产品视角

校园互助服务平台是一个面向高校学生群体的校内互助信息对接与任务协作Web网站系统。其建设背景源于当前校园内大量碎片化的互助需求——快递代取、二手交易、学习答疑、活动组队、失物招领等——主要依赖微信群、QQ群及朋友圈等非结构化渠道传播，导致信息分散易遗漏、供需匹配效率低下、交易过程缺乏安全保障与信用追溯机制。

本平台旨在构建一个实名可信、流程规范、信息聚合的校园互助一站式Web入口，通过将离散的求助与帮助行为线上化、流程化，为学生提供以下核心价值：

- **安全可信的用户体系**：以手机号注册为基础，学号认证为可选增强项，在确保用户身份真实可追溯的同时，尊重学生对匿名表达的诉求。
- **精准聚焦的互助场景**：覆盖快递跑腿、学习辅导、二手交易、组队搭伴、失物招领、拼车拼单等基本类型，解决学生痛点。
- **透明高效的流程管理**：提供从需求发布、接单沟通、状态跟踪到完成评价的完整闭环，替代即时通讯群内刷屏、手动接龙的原始模式。
- **可量化的信用参考**：通过互评机制积累用户信用评分，为交易双方提供决策依据，激励友善行为、约束违规操作。

从产品形态上看，本平台是一个**统一的Web网站系统**，根据用户角色的不同提供差异化的功能界面：

- **学生端（前台网站）**：面向全体在校学生，采用响应式Web设计，适配PC浏览器、平板及手机移动端访问。学生用户通过浏览器即可完成注册登录、发布需求、浏览接单、即时沟通、订单管理、信用评价等全部操作，无需下载安装任何客户端应用。
- **管理员端（后台网站）**：面向平台运营与管理人员，提供独立的后台管理界面。管理员通过Web浏览器登录后，可进行内容审核、用户账号管理、纠纷仲裁及数据统计分析等运营工作。

系统部署以单个高校为单位，支持多校区场景下按地理位置筛选任务，服务范围严格限定于本校学生群体内部。

| 用户类型                   | 特征描述                                                                                                                                                                                                                                                                                                                                                           | 主要需求                                                                                                                                                           |
| :------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **在校学生（需求发布者）** | 平台的核心用户群体，以本科生、研究生为主。具备以下特征：<br>1. 智能手机普及率高，习惯通过Web或移动浏览器访问各类服务，对下载专用App或依赖特定小程序平台意愿不强；<br>2. 有大量碎片化互助需求，追求便捷高效；<br>3. 对隐私保护有一定敏感度，部分用户对强制公开学籍信息存在抵触，期望在特定场景可匿名发声；<br>4. 分布于不同校区，有跨校区跑腿或拼车出行的实际诉求。 | 快速发布各类互助需求；<br>查看订单状态与历史记录；<br>与接单者在线沟通细节；<br>在特定分类下选择匿名发布；<br>按校区筛选或标注任务地点；<br>对完成的服务进行评价。 |

### 2.3 约束

本系统的设计、开发与部署受到以下限制。

**技术约束**

1. 系统须基于Web架构实现，采用前后端分离的开发模式。
2. 前端为响应式HTML/CSS/JavaScript应用，需兼容主流桌面浏览器及移动端浏览器，确保在不同尺寸屏幕上的可用性。
3. 后端服务提供统一的RESTful API，部署环境限定为Linux服务器操作系统。
4. 数据持久化采用MySQL 8.0及以上版本的关系型数据库，缓存服务采用Redis 6.0及以上版本。
5. 即时通讯功能须基于WebSocket协议实现实时双向消息推送，以支持供需双方的低延迟沟通。

**集成约束**

1. 为完成用户身份核验，系统需与所在高校的校园统一身份认证系统进行对接，具体对接协议视各校信息中心实际提供的接口规范而定。
2. 用户上传的图片等静态文件需存储至第三方对象存储服务，以降低服务器本地存储压力。
3. 用户注册、登录及关键操作环节的短信验证码发送功能需通过第三方短信服务平台提供的API实现。

**法律法规约束**

1. 平台涉及对用户个人信息的采集、存储和使用，包括但不限于手机号码、学号、姓名等，须严格遵守《中华人民共和国个人信息保护法》的相关规定，遵循合法、正当、必要与诚信原则。
2. 系统在收集个人信息前须取得用户的明示同意，并仅采集与平台服务直接相关的必要信息。
3. 用户有权申请注销其个人账号，系统须在注销完成后依法对用户数据进行删除或匿名化处理。
4. 系统须建立内容安全审核机制，对用户公开发布的需求内容进行敏感词过滤，必要时辅以人工审核，以履行网络运营者的内容管理责任。

### 2.4 假设

本系统的需求定义与后续开发工作建立在以下假设条件之上。

**用户设备与使用习惯假设**

假设目标高校的在籍学生均拥有可连接互联网的智能手机、平板或个人电脑设备，且日常使用主流Web浏览器访问各类网络服务。学生群体对通过网站而非独立App完成信息浏览、表单填写等操作具有较高的接受度。

**校园网络环境假设**

假设目标高校的校园范围内已实现无线网络和移动蜂窝网络的基本覆盖，学生用户在教室、宿舍、食堂、图书馆等主要活动区域均能稳定接入互联网，网络带宽与延迟水平足以支持网页的正常加载、图片上传及WebSocket实时消息的可靠传输。

**校园统一身份认证系统可用性假设**

假设各目标高校均部署有校园统一身份认证系统，并已对外开放必要的应用程序编程接口供校内第三方系统对接调用。 该认证系统的服务稳定可靠，能够及时响应本平台发起的学号与密码验证请求，且校方信息中心配合提供必要的接口文档与技术支持。

**校方支持与部署配合假设**

假设本平台作为校内服务项目，能够获得学校学生工作部门、信息化建设部门等相关管理机构的认可与支持。平台部署所需的服务器资源、域名备案、网络安全策略配置等环节能够得到校方或项目发起方的配合与保障，系统上线后亦能获得持续的基本运维支持。

**用户行为诚信假设**

假设绝大多数学生用户在完成实名核验后，能够本着诚实守信的原则使用平台各项功能，包括如实发布互助需求信息、按约定完成已接受的订单任务、客观公正地对合作方进行信用评价。对于少数恶意发布虚假信息、接单后失约或恶意差评等违规行为，假设可通过信用评分机制和管理员人工介入予以有效约束与纠正。

**第三方服务稳定性假设**

假设本平台所依赖的第三方服务，包括但不限于对象存储服务、短信平台服务，均能保持商业级别的服务可用性与稳定性，其应用程序编程接口的调用频率限制、计费模式及数据安全策略与本平台的预期使用规模相匹配，不会因第三方服务故障或策略变更导致平台核心功能不可用。

**法规政策延续性假设**

假设国家在个人信息保护、网络安全、互联网内容管理等方面的法律法规与监管要求在项目开发周期及上线初期运行阶段保持相对稳定，不出现因法规重大调整而导致系统核心架构或业务规则被迫进行颠覆性修改的情形。
