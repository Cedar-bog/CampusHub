## 1 引言

### 1.1 目的

本文档旨在对“校园互助服务平台”的软件需求进行完整、清晰的规格说明。该文档详尽说明了本产品的功能需求和非功能需求，是进行系统设计的基础，也是编写测试用例和进行系统测试的主要依据。同时，该文档也是用户确认软件功能需求的主要依据，将为项目管理人员、开发人员、测试人员及最终用户提供统一的参考标准。

### 1.2 文档约定

本文档遵循IEEE 830-1998《软件需求规格说明推荐实践》标准编写。文档中需求的优先级采用三级划分：

- **高（High）** ：关键任务需求，必须在本版本实现，软件验收的必要条件；
- **中（Medium）** ：支持必要的系统操作，如有必要可延迟到下一版本，但实现后能显著增强产品功能；
- **低（Low）** ：功能或质量上的增强，如资源允许可实现，可使产品更完善。

文档中“必须”“应”“可以”等措辞遵循RFC 2119规范。

### 1.4 产品的范围

本产品是一个面向大学生的校园互助服务平台，旨在整合校园内各类碎片化的互助需求，提供统一、高效、可信的信息对接渠道。平台支持学生发布和接单以下互助类型：

- 快递代取与跑腿代办
- 学习辅导与知识分享
- 二手闲置交易
- 组队匹配与活动搭伴

平台将取代传统的微信群、QQ群等分散的互助信息传播方式，通过实名认证、信用评价和订单追踪等机制，构建安全、透明、高效的校园互助生态。

## 2 综合描述

### 2.1 产品的前景

随着高校规模扩大和学生需求的多元化，校园内存在大量碎片化的互助需求，包括快递代取、二手交易、学习辅导、活动组队等。目前，这些需求主要依赖微信群、QQ群和朋友圈等渠道传播，存在信息分散易遗漏、供需匹配效率低、缺乏安全保障和信用机制等突出问题。

校园互助服务平台是一个全新的、自含型产品，旨在构建一个面向校园场景的“需求发布—精准匹配—安全交易—信用管理”一体化平台。该平台聚焦校园场景、构建实名认证体系，通过分类功能模块精准匹配校内需求，同时依托学校认证机制保障用户信任与信息安全，解决学生实际痛点。

### 2.2 产品的功能

本产品的主要功能概述如下（详细内容见第4节系统特性）：

1. **用户注册与实名认证**：支持学生通过校园统一身份认证进行注册，绑定学号和校园信息，确保用户身份真实可靠。
2. **需求发布与管理**：用户可发布多种类型的互助需求，标注类型、详情、时限、报酬等信息，支持对已发布需求的编辑和撤销。
3. **需求浏览与智能匹配**：用户可按类型、距离、报酬等条件筛选需求，系统根据用户标签和地理位置智能推荐匹配。
4. **接单与订单管理**：用户可接受他人发布的互助需求，系统跟踪订单状态（待接单/进行中/已完成/已取消）。
5. **即时通讯**：供需双方可通过内置聊天工具进行在线沟通，支持文本、图片消息传递。
6. **信用评价体系**：互助完成后双方互评，形成信用评分，激励良好行为、约束违规操作。
7. **管理员后台**：管理员可进行内容审核、用户管理、纠纷处理和数据统计分析。

### 2.3 用户类型和特征

本系统涉及以下三类用户：

| 用户类型                   | 特征描述                                                                     | 主要需求                                         |
| -------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------ |
| **在校学生（需求发布者）** | 平台的主要用户群体，需要发布各类互助需求；期望快速找到帮助者，且交易安全可靠 | 发布需求、查看订单状态、在线沟通、完成评价       |
| **在校学生（服务接单者）** | 愿意利用闲暇时间提供帮助，获取报酬或积分                                     | 浏览需求、接单、完成任务、获取收益、接收评价     |
| **平台管理员**             | 负责平台日常运维与监管的人员                                                 | 内容审核、用户管理、纠纷调解、数据统计、系统运维 |

所有学生用户必须通过校园统一身份认证完成实名注册，方可使用平台各项功能。

### 2.4 运行环境

**服务器端：**

- 操作系统：Linux（CentOS 7+ / Ubuntu 20.04+）
- Web服务器：Nginx 1.18+
- 应用服务器：Java 11+ / Spring Boot 2.x 或 Node.js 16+
- 数据库：MySQL 8.0+
- 缓存服务：Redis 6.0+

**客户端：**

- 微信小程序端：支持微信版本8.0及以上
- Web端：支持主流浏览器（Chrome 80+、Firefox 75+、Safari 13+、Edge 80+）
- 移动端H5：适配iOS 13+和Android 8.0+系统

**网络环境：**

- 需接入校园内网，支持校园统一身份认证系统的调用
- 支持HTTPS安全传输协议

### 2.5 设计和实现上的限制

1. **实名认证依赖**：学生用户的注册和登录必须通过所在高校的统一身份认证系统，因此平台的部署需要与各高校的信息中心对接。
2. **校园地理范围限制**：平台的服务范围限定在单个校园内部，跨校互助暂不支持。
3. **法律法规遵从**：平台涉及用户个人信息采集和处理，必须严格遵守《个人信息保护法》等相关法律法规。
4. **开发周期限制**：第一版应在3个月内完成核心功能的开发与上线测试。
5. **多端适配要求**：系统需同时支持微信小程序、Web端和移动端H5，后端API需统一设计。

### 2.6 假设和依赖

1. **假设用户拥有智能手机**：假设目标用户（大学生）均拥有智能手机，并可正常使用微信。
2. **假设校园网络覆盖良好**：假设校园内Wi-Fi和移动网络覆盖良好，用户可随时接入平台。
3. **依赖校园统一认证系统**：系统的实名认证功能依赖于高校统一身份认证系统的开放接口。
4. **依赖第三方支付平台**：涉及有偿互助时，在线支付功能依赖于微信支付等第三方支付接口。
5. **依赖微信小程序平台**：小程序端的运行依赖于微信平台的审核和发布机制。

## 3 外部接口需求

### 3.1 用户界面

**3.1.1 微信小程序端**

- 首页：展示功能入口图标（快递代取、学习辅导、二手交易、组队匹配等）、轮播公告、推荐需求列表、底部导航栏。
- 需求大厅页：按分类展示需求卡片，支持筛选（类型、距离、报酬等）和搜索。
- 发布需求页：提供需求类型选择、详情填写表单（标题、描述、时限、地点、报酬等）、图片上传。
- 消息页：展示聊天会话列表，支持即时通讯。
- 个人中心页：展示用户信息、信用评分、我的需求、我的接单、订单记录、评价管理、设置。

**3.1.2 Web管理后台**

- 登录页：管理员账号密码登录，支持二次验证。
- 仪表盘：展示平台关键数据（今日新增需求数、订单完成率、用户增长等）的可视化图表。
- 用户管理页：用户列表、实名认证审核、账号状态管理。
- 内容审核页：需求内容的审核与管理（通过/驳回/删除）。
- 纠纷处理页：投诉举报的查看与处理。
- 数据统计页：按时间维度统计各类互助数据，支持报表导出。

**3.1.3 界面设计要求**

- 界面风格简洁清新，符合大学生审美。
- 配色以浅色调为主，辅以品牌主色调。
- 交互操作流畅，重要操作提供二次确认。
- 支持响应式布局，适配不同尺寸屏幕。

### 3.2 硬件接口

本系统不涉及特殊硬件设备，无需定义专用硬件接口。

### 3.3 软件接口

| 接口类型             | 接口说明                 | 用途                 |
| -------------------- | ------------------------ | -------------------- |
| 校园统一身份认证接口 | 调用高校CAS/OAuth2.0接口 | 学生实名认证与登录   |
| 微信登录接口         | 微信开放平台API          | 小程序端用户授权登录 |
| 微信支付接口         | 微信支付API v3           | 有偿互助的在线支付   |
| 消息推送接口         | 微信订阅消息API          | 订单状态变更提醒     |
| 文件存储接口         | OSS/COS对象存储API       | 用户上传图片的存储   |
| 短信服务接口         | 第三方短信平台API        | 重要操作的短信验证   |

### 3.4 通信接口

- 客户端与服务器之间采用HTTPS协议进行加密通信。
- 后端服务之间的内部调用采用HTTP/RESTful API。
- 即时通讯功能采用WebSocket协议，实现实时消息推送。
- 数据交换格式统一采用JSON。

## 4 系统特性

### 4.1 用户注册与认证模块

**4.1.1 说明和优先级**

该模块实现用户注册、登录和实名认证功能，是整个平台安全可信的基础。优先级：高。

**4.1.2 激励/响应序列**

1. 用户打开小程序，点击“登录/注册”。
2. 系统调用微信授权登录接口，获取用户OpenID。
3. 用户选择所在学校，系统跳转至校园统一身份认证页面。
4. 用户输入学号和密码完成认证，系统获取学号、姓名等实名信息。
5. 系统验证成功后，创建用户账号并返回登录态。
6. 用户进入平台首页。

**4.1.3 功能需求**

| 编号  | 功能需求                                             | 优先级 |
| ----- | ---------------------------------------------------- | ------ |
| REG-1 | 系统应支持通过微信授权登录                           | 高     |
| REG-2 | 系统应支持通过校园统一身份认证完成实名注册           | 高     |
| REG-3 | 系统应验证学号的唯一性，一个学号仅能注册一个账号     | 高     |
| REG-4 | 系统应存储用户的加密身份信息（学号、姓名、院系等）   | 高     |
| REG-5 | 系统应支持用户编辑个人资料（头像、昵称、联系方式等） | 中     |
| REG-6 | 系统应提供账号注销功能，注销后按法规处理用户数据     | 中     |

### 4.2 需求发布与管理模块

**4.2.1 说明和优先级**

该模块支持用户发布各类互助需求，并对已发布需求进行管理。优先级：高。

**4.2.2 激励/响应序列**

1. 用户点击“发布需求”按钮。
2. 系统展示需求类型选择页（快递代取/学习辅导/二手交易/组队匹配）。
3. 用户选择类型后进入表单填写页，填写标题、描述、时限、地点、报酬等信息。
4. 用户可选择上传图片（最多9张），点击“发布”。
5. 系统对内容进行敏感词检测，通过后创建需求订单。
6. 系统返回发布成功提示，需求进入“待接单”状态并展示在需求大厅。

**4.2.3 功能需求**

| 编号  | 功能需求                                                                     | 优先级 |
| ----- | ---------------------------------------------------------------------------- | ------ |
| PUB-1 | 系统应支持用户按类型发布需求，包括快递代取、学习辅导、二手交易、组队匹配四类 | 高     |
| PUB-2 | 系统应提供需求表单，包含标题、描述、时限、地点、报酬等必填项                 | 高     |
| PUB-3 | 系统应支持上传图片（最多9张），自动压缩后存储                                | 中     |
| PUB-4 | 系统应对发布内容进行敏感词过滤和审核                                         | 高     |
| PUB-5 | 系统应支持用户查看自己发布的所有需求及对应状态                               | 中     |
| PUB-6 | 系统应支持用户在无人接单时编辑或撤销已发布需求                               | 中     |
| PUB-7 | 发布快递代取需求时，系统应支持填写快递取件码/快递点信息                      | 中     |

### 4.3 需求浏览与接单模块

**4.3.1 说明和优先级**

该模块支持用户浏览平台上的互助需求，筛选和搜索，并选择接受他人发布的需求。优先级：高。

**4.3.2 激励/响应序列**

1. 用户进入需求大厅，系统默认按发布时间倒序展示需求列表。
2. 用户可按需求类型、距离远近、报酬等条件筛选需求。
3. 用户点击需求卡片查看详情，确认后点击“接单”。
4. 系统验证用户身份后锁定订单，需求状态变更为“进行中”。
5. 系统建立供需双方的即时通讯会话，双方可在线沟通细节。

**4.3.3 功能需求**

| 编号  | 功能需求                                                                           | 优先级 |
| ----- | ---------------------------------------------------------------------------------- | ------ |
| BRO-1 | 系统应以卡片列表形式展示需求大厅，每张卡片包含需求标题、类型、报酬、距离、发布时间 | 高     |
| BRO-2 | 系统应支持按需求类型进行筛选                                                       | 高     |
| BRO-3 | 系统应支持按距离远近排序（基于用户当前位置和需求地点）                             | 中     |
| BRO-4 | 系统应支持关键词搜索需求                                                           | 中     |
| BRO-5 | 系统应提供需求详情页，展示完整信息及发布者信用评分                                 | 高     |
| BRO-6 | 用户点击接单后，系统应立即锁定订单，防止重复接单                                   | 高     |
| BRO-7 | 系统应支持接单者在特定条件下取消接单（需与发布者协商）                             | 中     |

### 4.4 订单管理与追踪模块

**4.4.1 说明和优先级**

该模块管理互助订单的全生命周期状态，追踪订单进度，并支持订单完成确认。优先级：高。

**4.4.2 激励/响应序列**

1. 接单成功后，订单进入“进行中”状态，双方可在订单详情页查看进度。
2. 接单者完成任务后点击“完成互助”，系统通知发布者确认。
3. 发布者收到通知后检查完成情况，点击“确认完成”。
4. 系统将订单状态变更为“已完成”，进入评价流程。
5. 若涉及报酬，系统触发支付结算流程。

**4.4.3 功能需求**

| 编号  | 功能需求                                           | 优先级 |
| ----- | -------------------------------------------------- | ------ |
| ORD-1 | 系统应追踪订单状态：待接单、进行中、已完成、已取消 | 高     |
| ORD-2 | 系统应在订单状态变更时向相关用户推送通知           | 高     |
| ORD-3 | 系统应支持接单者标记任务完成                       | 高     |
| ORD-4 | 系统应支持发布者确认任务完成                       | 高     |
| ORD-5 | 系统应记录订单操作日志（接单时间、完成时间等）     | 中     |
| ORD-6 | 系统应支持订单取消，需双方同意或由管理员介入       | 中     |
| ORD-7 | 系统应提供订单历史记录查询功能                     | 中     |

### 4.5 即时通讯模块

**4.5.1 说明和优先级**

该模块提供供需双方在线沟通功能，支持文本和图片消息。优先级：中。

**4.5.2 激励/响应序列**

1. 接单成功后，系统自动创建供需双方的聊天会话。
2. 用户可从订单详情页或消息列表进入聊天界面。
3. 用户发送消息，系统实时推送至对方。
4. 对方收到消息推送，进入聊天界面查看并回复。

**4.5.3 功能需求**

| 编号   | 功能需求                       | 优先级 |
| ------ | ------------------------------ | ------ |
| CHAT-1 | 系统应支持供需双方发送文本消息 | 中     |
| CHAT-2 | 系统应支持发送图片消息         | 中     |
| CHAT-3 | 系统应展示消息已读/未读状态    | 低     |
| CHAT-4 | 系统应支持查看历史聊天记录     | 中     |
| CHAT-5 | 系统应对敏感词汇进行过滤       | 高     |

### 4.6 信用评价体系模块

**4.6.1 说明和优先级**

该模块在互助完成后供双方互评，形成用户信用评分，构建平台信任体系。优先级：中。

**4.6.2 激励/响应序列**

1. 订单确认完成后，系统引导双方进行互评。
2. 用户选择评分（1-5星）并可选填写文字评价。
3. 系统计算并更新用户的综合信用评分。
4. 评价内容在用户个人主页展示。

**4.6.3 功能需求**

| 编号   | 功能需求                                           | 优先级 |
| ------ | -------------------------------------------------- | ------ |
| EVAL-1 | 系统应支持互助完成后双方互评（1-5星）              | 中     |
| EVAL-2 | 系统应支持用户填写文字评价（可选）                 | 中     |
| EVAL-3 | 系统应基于历史评价计算用户综合信用评分并公开展示   | 中     |
| EVAL-4 | 系统应允许用户查看自己收到的所有评价               | 中     |
| EVAL-5 | 信用评分应影响用户在平台上的部分权限（如接单限制） | 低     |

### 4.7 管理员后台模块

**4.7.1 说明和优先级**

该模块为平台管理员提供内容审核、用户管理和数据分析功能。优先级：高。

**4.7.2 激励/响应序列**

1. 管理员通过Web端登录后台，系统验证账号权限。
2. 管理员进入仪表盘，查看平台运行数据概览。
3. 管理员可对用户举报的内容进行审核处理。
4. 管理员可对违规用户进行警告、限制或封禁。
5. 管理员可查看数据统计报表。

**4.7.3 功能需求**

| 编号    | 功能需求                                                   | 优先级 |
| ------- | ---------------------------------------------------------- | ------ |
| ADMIN-1 | 系统应提供管理员Web后台登录功能，支持账号密码及二次验证    | 高     |
| ADMIN-2 | 系统应支持管理员审核用户发布的需求内容（通过/驳回/删除）   | 高     |
| ADMIN-3 | 系统应支持管理员管理用户账号（查看、冻结、解冻）           | 高     |
| ADMIN-4 | 系统应支持管理员处理用户举报和纠纷投诉                     | 中     |
| ADMIN-5 | 系统应提供数据统计仪表盘，展示需求发布量、完成率等关键指标 | 中     |
| ADMIN-6 | 系统应支持管理员查看和导出运营数据报表                     | 低     |

## 5 其他非功能需求

### 5.1 性能需求

| 编号  | 性能指标     | 目标值                                |
| ----- | ------------ | ------------------------------------- |
| PER-1 | 页面加载时间 | 首屏加载时间不超过3秒                 |
| PER-2 | 接口响应时间 | API接口95%的请求响应时间不超过500ms   |
| PER-3 | 并发支持能力 | 系统应支持不少于500名用户同时在线操作 |
| PER-4 | 消息推送延迟 | 订单状态变更通知推送延迟不超过5秒     |
| PER-5 | 系统可用性   | 系统可用性不低于99.5%（月度）         |

### 5.2 安全性需求

| 编号  | 安全需求                                               | 优先级 |
| ----- | ------------------------------------------------------ | ------ |
| SEC-1 | 用户身份信息（学号、姓名等）必须加密存储，不得明文保存 | 高     |
| SEC-2 | 所有网络通信必须使用HTTPS加密传输                      | 高     |
| SEC-3 | 用户敏感操作（如支付、修改密码）需进行二次验证         | 高     |
| SEC-4 | 系统应具备防SQL注入、XSS攻击、CSRF攻击的能力           | 高     |
| SEC-5 | 用户数据应遵循最小采集原则，仅收集平台运行必需的信息   | 高     |
| SEC-6 | 系统应记录关键操作日志，支持安全审计追溯               | 中     |

### 5.3 软件质量属性

| 质量属性 | 要求说明                                                           |
| -------- | ------------------------------------------------------------------ |
| 可用性   | 界面简洁直观，新用户无需培训即可完成基本操作；重要功能提供引导提示 |
| 可靠性   | 系统应保证数据一致性，订单状态变更不会因网络异常而出现脏数据       |
| 可维护性 | 系统采用模块化设计，各模块之间低耦合，便于独立维护和升级           |
| 可扩展性 | 系统架构应支持后续增加新的互助类型和功能模块，无需大规模重构       |
| 可移植性 | 后端服务应可在主流Linux发行版上运行，数据库应支持主流关系型数据库  |

### 5.4 业务规则

| 编号   | 业务规则     | 说明                                                                   |
| ------ | ------------ | ---------------------------------------------------------------------- |
| RULE-1 | 实名认证规则 | 所有学生用户必须通过校园统一身份认证方可使用平台功能                   |
| RULE-2 | 校园范围限制 | 平台服务范围限于单个校园，暂不支持跨校互助                             |
| RULE-3 | 内容审核规则 | 用户发布的需求内容需经过敏感词过滤，违规内容将被系统拦截或人工审核     |
| RULE-4 | 信用评分规则 | 用户信用评分基于历史订单完成率和互评分数综合计算，低信用分用户可能受限 |
| RULE-5 | 纠纷处理规则 | 发生交易纠纷时，用户可向管理员申诉，管理员有权介入调解并作出最终裁决   |
| RULE-6 | 报酬结算规则 | 涉及报酬的订单，报酬在双方确认完成后方可结算                           |

### 5.5 用户文档

系统交付时需提供以下用户文档：

1. **用户使用手册**：面向学生用户的完整操作指南，包含注册登录、发布需求、接单、评价等全流程说明。
2. **管理员操作手册**：面向平台管理员的运维管理指南，包含内容审核、用户管理、数据查看等操作说明。
3. **常见问题解答（FAQ）** ：整理常见问题及解决方法，内置在小程序帮助中心。
